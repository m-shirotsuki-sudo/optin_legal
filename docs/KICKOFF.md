# 契約書作成ツール — KICKOFF 仕様書（完成版）

最終更新: 2026-06-11 / 宛先: Claude Code / 作成: PMO（壁打ち結果の確定版）

---

## 0. このドキュメントの位置づけ

OPT-IN HD 全社で使う「契約書作成ツール」を新規開発するための確定仕様。
あわせて参照実装として **`contract-demo.html`**（Men's Rise VIPプラン 20260519版の動作デモ）と、
**原本docx（Men's Rise 20260519版）** を同梱する。

**体裁・改ページ・署名欄の詰め・第4条の支払い欄の挙動は、すべてデモに実装済み。**
デモを「正」とし、これを会社×プランを選べる汎用ツールへ一般化するのがゴール。

---

## 1. 背景と解決したい課題

- 現状、セールスが原本docxを手でコピーして可変箇所を埋め、クラウドサインで締結している。
- 事務が不得手なセールスがWordを自由編集するため、文字ずれ・全角半角混在・条文破損・
  空欄放置が発生し、会社の信用を損なっている。
- 根本原因は「セールスにWordのフル編集権限が常時ある」こと。
  → **可変箇所だけを入力させ、本文は構造的に触れないツール化**で解決する。

---

## 2. 何を作るか（1段落サマリー）

セールス担当が「会社」と「プラン」を選び、案件ごとの可変項目（顧客名・住所・電話・
締結日・第4条の支払条件など）だけを入力すると、**本文の文言が固定された契約書PDF**が
発行されるWebツール。発行PDFはダウンロードし、人がクラウドサインへ手動アップして締結する。
原本docx・金額・口座・本文テンプレートの管理は管理者（PMO）のみ。

---

## 3. 絶対に守る原則（最重要）

1. **可変箇所以外のすべてが原本ママ保持。**
   - 変えてよいのは可変箇所（顧客名・住所・電話・締結日・第4条支払条件・契約担当者名）だけ。
   - **黒い条文・金額・口座・サービス表・赤字条項・赤枠注意文、すべて一字一句固定。**
   - 「赤字＝原本ママ」ではない。**赤も黒も含め本文全体が原本ママ**。赤字は「色も原本どおり保持」の意。
2. **色も原本どおり保持。**
   - 冒頭の特商法注意文（赤枠＋赤文字）、第9条クーリングオフ（赤文字）は赤のままPDFに出す。
3. **文言の正しさは原本との機械照合で担保する。**（刷新事故防止の要・第7章）
4. **権限分離。** セールス＝入力／PDF生成のみ。管理者＝原本／金額／口座／テンプレ管理。

---

## 4. 技術スタック

既存の invoice 管理システムに揃える。
- フロント／サーバー: **Next.js**（App Router）
- DB／認証／ストレージ: **Supabase**（Postgres + Auth + Storage）
- デプロイ: **Vercel**
- PDF生成: **HTML→PDF**。サーバー側で **Puppeteer**（or Playwright）。
  `format: 'A4'`, `printBackground: true`, `@media print` を効かせる。
- 認証: Supabase Auth。ロールは `sales` / `admin`。

> HTML→PDF採用の根拠: docxは体裁が壊れやすく刷新時に事故る（実際、原本は全角半角混在・
> 文字ずれ・テキストボックス内の赤文字取りこぼしがあった）。HTMLなら本文をコードとして
> 一元管理でき、刷新はテンプレ1枚を直すだけ。差分はGitで履歴化できる。

## 5. データモデル（Supabase）

### `companies`（会社マスタ）
| カラム | 型 | 説明 |
|---|---|---|
| id | uuid PK | |
| code | text | 例: ME01, GH02, AI01 など |
| name | text | 表示名（株式会社Men'sRise 等） |
| seller_info | jsonb | 乙情報（住所・法人名・代表者・電話）固定値 |

### `plans`（プランマスタ）
| カラム | 型 | 説明 |
|---|---|---|
| id | uuid PK | |
| company_id | uuid FK | |
| name | text | 例: VIPプラン |
| version | text | 例: 20260519（刷新ごとに新バージョン行を追加） |
| is_active | bool | 現行版フラグ（旧版は false にして履歴保持） |
| template_html | text | **PDF生成用の本文HTMLテンプレート（{{var}}入り）** |
| constants | jsonb | 金額・口座・サービス内容など固定値 |
| variable_fields | jsonb | 可変項目の定義（第6章） |
| original_docx_path | text | **原本docxの保管場所**（Supabase Storage パス or Drive URL） |
| original_checksum | jsonb | 原本から抽出した照合用キーフレーズ群（第7章） |

> **刷新の扱い**: 改訂時は同一planの version 違いを新規行で追加し、旧行を is_active=false に。
> 過去の締結時にどの版だったかを追えるようにする。

### `contracts`（発行履歴・推奨）
| カラム | 型 | 説明 |
|---|---|---|
| id | uuid PK | |
| plan_id | uuid FK | どの会社／プラン／版で発行したか |
| created_by | uuid | 発行したセールス |
| input_values | jsonb | 入力された可変値 |
| pdf_path | text | 生成PDFの保管先 |
| created_at | timestamptz | |

## 6. 可変項目の定義（Men's Rise VIPプランの確定版）

`variable_fields` は以下。**第4条の支払条件は併用可・空欄許容**が肝。

```json
[
  {"key":"customer_name","label":"氏名","type":"text","required":true},
  {"key":"customer_addr","label":"住所","type":"textarea","required":true},
  {"key":"customer_tel","label":"電話番号","type":"tel","pattern":"[0-9-]+","required":true},
  {"key":"contract_date","label":"締結年月日","type":"date","required":true},
  {"key":"staff_name","label":"契約担当者氏名","type":"text","required":true},

  {"group":"第4条 支払条件（該当欄のみ入力・併用可・空欄は空欄のまま出力）","fields":[
    {"key":"debit_times","label":"デビットカード 支払回数（回）","type":"number","required":false},
    {"key":"shop_times","label":"ショッピングクレジット 支払回数（回）","type":"number","required":false},
    {"key":"credit_company","label":"クレジット会社名（Mosh / CBS / Lifety 等・手入力）","type":"text","required":false},
    {"key":"debit_date","label":"引落 年月日（2026年〇月〇日）","type":"text","required":false},
    {"key":"bank_amount","label":"銀行振込額（円）","type":"text","required":false},
    {"key":"first_amount","label":"初回額（円・税込）","type":"text","required":false},
    {"key":"next_amount","label":"翌月額（円・税込）","type":"text","required":false}
  ]}
]
```

### 第4条 支払条件の決め事（重要・確定）
- 原本の信販表は**1つの表**。銀行振込・分割（初回／翌月）の金額欄が同居しており、**併用前提**。
- **排他選択にしない。** 「銀行 or 信販」の二択や、片方を消す挙動は禁止（初期実装の誤りを是正済み）。
- 7つの手入力欄（上記）を用意。**使う欄だけ埋め、未入力欄は契約書でも空ブランクのまま出力**する。
  （原本が手書き前提のブランクを持つため、空欄表示が正しい挙動）
- **クレジット会社名は手入力。** 原本に「Mosh」と印字されていたが、実際は CBS・Lifety 等も
  使うため固定をやめ、可変欄にする。**これは条文（権利義務）の改変ではなく、案件ごとに
  変わるべき記入項目を固定印字から正しく可変化する修正**。条文本体の文言は一切変えない。
- 銀行口座情報（三井住友・トランクNORTH支店・口座番号・名義）は固定値で常時表示。

## 7. 原本照合（刷新事故を防ぐ仕組み・必須）

管理者が `template_html` を保存／更新するとき、サーバー側で次を実行する。

1. `original_docx_path` の原本docxからテキストを抽出する。
   - **必ず `word/document.xml` を直読みし、`<w:t>` を連結する方式にする。**
   - **テキストボックス（`<w:txbxContent>`）内のテキストも必ず拾う**こと。
     赤枠注意文・第9条クーリングオフは図形（テキストボックス）内にあり、
     通常のテキスト抽出（extract-text/mammoth）では**取りこぼす**。これが実際に事故った箇所。
2. 原本から主要キーフレーズ（特商法文言・コース名・各金額・サービス内容・条番号・
   クーリングオフ各項・中途解約返金規定・乙情報など）を抽出し、`template_html` のテキスト
   （HTMLタグと {{var}} を除去したもの）に**すべて含まれるか照合**する。
3. **不一致があれば保存をブロックし、差分を表示**。一致したら `original_checksum` に保存。

> このロジックはデモ作成時に実証済み（document.xml直読み → 16キーフレーズ照合 → NG0件で確認）。
> 全社運用では、契約書が版改訂されるたびにこの照合を必須ゲートにする。

## 8. 画面構成

### A. セールス画面（ロール: sales）
1. **会社を選ぶ**（companies プルダウン）
2. **プランを選ぶ**（選んだ会社の is_active な plans）
3. 選択でそのプランの金額・サービス内容が**表示のみ**で出る（編集不可）
4. **可変項目フォーム**（variable_fields から動的生成）
   - 必須チェック・形式チェック（電話番号など）
   - 第4条支払条件は7欄すべて任意入力・併用可
5. **右側にリアルタイムプレビュー**（デモと同じ。黄色＝可変箇所、赤＝原本ママ）
6. **「PDFを発行」ボタン** → サーバーでPDF生成 → ダウンロード
   - （将来）クラウドサインAPI送信ボタンを追加

### B. 管理画面（ロール: admin のみ）
1. 会社の登録・編集（乙情報など）
2. プランの登録・編集
   - 原本docxのアップロード（Supabase Storageへ保管）
   - 本文HTMLテンプレートの編集
   - 金額・口座・サービス内容（constants）の編集
   - 可変項目（variable_fields）の定義
3. **原本照合チェック**（テンプレ保存時に自動実行・第7章）
4. プランの版管理（刷新時に新バージョン追加、旧版を非アクティブ化）

## 9. PDF体裁要件（デモに実装済み・必ず踏襲）

`contract-demo.html` のCSSをテンプレートの基礎とする。以下は維持すること。

- A4縦、明朝体（Yu Mincho系）、本文 約9.8pt、行間1.72
- **赤枠注意文**: `border:1.5px solid #FF0000; color:#FF0000`
- **赤字条項**: クーリングオフ等は `color:#FF0000` を保持
- **改ページ制御（表＋項単位で割れない）**:
  - 表: `break-inside: avoid`
  - 各項(段落): `break-inside: avoid`
  - 見出し: `break-after: avoid`（見出しだけ末尾に取り残さない）
  - 赤字クーリングオフは「自然に流れてOK」（ページ頭固定はしない）
- **可変箇所ハイライト**: 画面では黄色（`.var`）、**印刷時は背景を消す**
  （`@media print` で `.var{background:transparent}`）
- **未入力の可変欄**: 画面では「（未入力）」表示、**印刷時は空ブランク**で出す
- **署名欄は詰める**: line-height 1.4、行margin最小、氏名と「印」は同一行
- **第4条**: 銀行口座は常時表示、信販表の各ブランクに入力値を差し込む（第6章の決め事）

## 付録: 確定した設計判断（壁打ち結果サマリー）
- 原本の扱い: docxは**保管・参照用**に登録。PDF生成用本文は**管理者が整備**（自動取り込みはしない）。
- クラウドサイン: まず**手動アップで運用開始**、API連携は後付け。
- 権限: **セールス＝入力／生成のみ、管理者＝原本／金額／口座／テンプレ管理**。
- 第4条支払い: **排他選択せず7欄手入力・併用可・空欄許容・会社名は手入力**。
- 原本ママ保持: **可変箇所以外は赤も黒もすべて固定**。色も原本どおり。
